# みんなの保健室ひだまり 相談予約システム 設計書

最終更新：2026年6月

---

## 1. サービス概要

| 項目 | 内容 |
|------|------|
| サービス名 | みんなの保健室ひだまり 相談予約システム |
| 運営 | みんなの保健室ひだまり |
| 目的 | 会社員が保健師との相談日時をオンラインで申し込めるようにする |
| 提供形態 | Webアプリ（スマートフォン・PCに対応） |

### 対象ユーザー

| ユーザー種別 | 説明 |
|------------|------|
| **予約者**（会社員） | ログイン不要で相談日時を申し込める |
| **管理者**（保健師・運営スタッフ） | ログインして予約管理・設定変更を行う |

## 2. システム構成

### 技術スタック

| 役割 | 技術 | 選定理由 |
|------|------|----------|
| Webアプリ基盤 | Next.js 16 | Vercelと最も相性が良く、将来の機能追加に対応しやすい |
| データベース・認証 | Supabase（PostgreSQL） | データベースと管理者認証が一体化しており、将来のユーザー認証追加も容易 |
| UIデザイン | Tailwind CSS + shadcn/ui | 品質の高いコンポーネントを素早く使えるライブラリ |
| ホスティング | Vercel | Next.jsとの公式連携、無料枠あり |

### 構成図

```
[ブラウザ（スマートフォン・PC）]
        ↕ HTTPS
[Vercel - Webアプリサーバー]
  ├─ 公開ページ（予約フォーム）
  └─ 管理画面（要ログイン）
        ↕ HTTPS API
[Supabase]
  ├─ PostgreSQL データベース
  │   ├─ 予約データ（bookings）
  │   ├─ スロット設定（available_slots）
  │   └─ システム設定（system_settings）
  └─ 認証サービス（管理者ログイン）
```

### データの流れ

- **予約申し込み時**：予約者がフォームに入力 → Vercelサーバーが受け取り → Supabaseの `bookings` テーブルに保存
- **管理者ログイン時**：管理者がメール・パスワードを入力 → Supabaseが認証 → セッションCookieが発行され管理画面にアクセス可能になる
- **設定変更時**：管理者が設定画面で変更 → Vercelサーバーが受け取り → Supabaseの `system_settings` テーブルに保存 → 予約フォームに即時反映

## 3. 画面一覧と機能仕様

### 予約者向け画面

| 画面 | URL | 機能概要 |
|------|-----|----------|
| 予約フォーム | `/` | 日付・時間の選択、名前・連絡先の入力、送信 |
| 予約完了 | `/booking/complete` | 予約受付完了のメッセージ、入力内容の確認 |

#### 予約フォームの操作フロー

1. 日付カルーセル（横スクロール）から希望日を選ぶ
2. 選択した日の予約可能な時間帯が表示される → 時間を選ぶ
3. 仮の名前（本名でなくても可）と連絡先（電話番号 または メールアドレス）を入力
4. 「予約を申し込む」ボタンで送信 → 完了画面へ

#### 日程選択のルール

- 本日からN日後以降の日付のみ選択できる（Nは管理者が設定、デフォルト：5日後）
- 固定時間帯モードの場合、管理者が指定した曜日の日付のみ表示される
- すでに予約が入っている時間帯は表示されない（1スロットにつき1予約のみ受け付ける）

### 管理者向け画面

| 画面 | URL | 機能概要 |
|------|-----|----------|
| ログイン | `/admin/login` | メールアドレス・パスワードでログイン |
| 予約一覧 | `/admin/bookings` | 予約の一覧表示・検索、ステータス変更、日時変更、メモ記入 |
| スロット管理 | `/admin/slots` | カスタムモード時のみ表示。個別の予約可能日時を追加・削除 |
| システム設定 | `/admin/settings` | 受付最短日数・スロットモード・時間帯・曜日の設定変更 |

## 4. 予約ステータスの仕様

予約には5つのステータスがある。初期状態は「仮予約」で、管理者が任意に変更できる。

| ステータス | 説明 | 日時変更 |
|----------|------|---------|
| **仮予約** | 申し込み直後の初期状態 | 変更可 |
| **確定** | 担当者が確認・連絡済みで日時が確定した状態 | 変更不可 |
| **キャンセル** | 予約をキャンセルした状態 | 変更不可 |
| **相談済** | 相談が完了した状態 | 変更不可 |
| **無断欠席** | 予定していた日時に来なかった状態 | 変更不可 |

**日時変更ルール：** ステータスが「仮予約」の間のみ、管理者が日時を変更できる。「確定」以降は変更不可。

---

## 5. スロット管理モードの仕様

管理者はシステム設定で、予約可能な時間帯の管理方法を2つのモードから選択できる。

### 固定時間帯モード

毎週同じ曜日・時間帯で受け付ける場合に使用する。

- **曜日指定**：日〜土から受付可能な曜日を複数選択できる（例：月・水・金のみ）
- **時間帯指定**：受付可能な時刻を複数登録できる（例：10:00、14:00、16:00）
- 指定した曜日のすべての日付に、同じ時間帯が適用される
- すでに予約が入った時間帯は自動的に非表示になる

**使用例：** 毎週火曜・木曜の10:00と15:00のみ受け付けたい場合

### カスタムモード

特定の日だけ受け付けたい場合や、不定期な対応に使用する。

- 管理者が「スロット管理」画面で日付と時刻を1件ずつ登録する
- 登録されたスロットのみが予約フォームに表示される
- 不要なスロットは削除できる

**使用例：** 来週の水曜14:00と、再来週の月曜10:00だけ受け付けたい場合

## 6. データベース設計

### テーブル一覧

| テーブル名 | 用途 |
|----------|------|
| `system_settings` | システム全体の設定を保存（常に1行のみ存在） |
| `available_slots` | カスタムモード用の個別スロット一覧 |
| `bookings` | 申し込まれた予約データ一覧 |

### system_settings テーブル（システム設定）

システム全体に1行だけ存在する設定テーブル。管理者がシステム設定画面から変更する。

| 項目名 | データ型 | 説明 | 初期値 |
|--------|---------|------|--------|
| 受付最短日数 | 整数 | 何日後から予約できるか | 5 |
| スロットモード | テキスト | 「固定（fixed）」または「カスタム（custom）」 | fixed |
| 固定時間帯リスト | テキスト配列 | 固定モード時の受付時刻（複数） | 10:00, 11:00, 14:00, 15:00 |
| 予約可能曜日 | 整数配列 | 固定モード時の受付曜日（0=日, 1=月, ..., 6=土） | 1,2,3,4,5（月〜金） |

### available_slots テーブル（スロット管理）

カスタムモード時のみ使用。管理者が手動で登録した予約可能日時の一覧。

| 項目名 | データ型 | 説明 |
|--------|---------|------|
| 日付 | 日付型 | 受付する日（例：2026-07-10） |
| 時刻 | 時刻型 | 受付する時刻（例：14:00） |
| 有効フラグ | 真偽値 | 無効にすることで一時的に非表示にできる |
| 登録日時 | 日時型 | スロットを登録した日時（自動記録） |

> 同じ日時の重複登録はできない（データベースレベルで防止）。

### bookings テーブル（予約データ）

予約者が申し込んだデータを保存するメインテーブル。

| 項目名 | データ型 | 説明 |
|--------|---------|------|
| 予約日 | 日付型 | 相談を希望する日付 |
| 予約時刻 | 時刻型 | 相談を希望する時刻 |
| お名前 | テキスト | 仮の名前でも可 |
| 連絡先 | テキスト | 電話番号またはメールアドレス |
| 連絡先種別 | テキスト | 「電話（phone）」または「メール（email）」 |
| ステータス | テキスト | 仮予約 / 確定 / キャンセル / 相談済 / 無断欠席 |
| メモ | テキスト | 管理者のみ閲覧・編集可能。連絡経緯・相談内容のメモ |
| 受付日時 | 日時型 | 申し込みが届いた日時（自動記録） |

### データの保存とアクセス制御

Supabaseの「Row Level Security（行レベルセキュリティ）」により、テーブルごとにアクセスを制限している。

| テーブル | 予約者（未ログイン） | 管理者（ログイン済み） |
|---------|------------|------------|
| system_settings | 読み取りのみ（予約フォームの表示に必要） | 読み取り・更新 |
| available_slots | 読み取りのみ（予約フォームの表示に必要） | 全操作（追加・削除・更新） |
| bookings | **書き込みのみ**（予約の新規登録のみ） | 全操作（一覧閲覧・更新・削除） |

> **設計上の重要な考え方：** 予約者は自分が登録した予約データを後から読み取ることができない。名前・連絡先などの個人情報は管理者のみが閲覧できる設計になっている。

## 7. 管理者認証の仕様

- Supabaseの認証機能を使用している
- **メールアドレス＋パスワード**でログインする
- ログイン状態はブラウザのCookieで管理される。ブラウザを閉じても維持される
- `/admin/` 配下のすべてのページはログインが必須。未ログインの場合はログイン画面に自動的に移動する
- すでにログイン済みの状態でログイン画面にアクセスすると、予約一覧画面に自動的に移動する
- 管理者アカウントの作成・削除はSupabaseのダッシュボードで行う

---

## 8. 管理者メモ・備考欄の仕様

- 各予約に対して、管理者だけが編集できるメモ欄がある
- 予約一覧画面の ✏️ボタンをクリックすると詳細モーダルが開き、メモを入力・保存できる
- メモの先頭50文字が予約一覧の「備考」列に表示される（スマートフォン表示ではカード内に全文表示）
- 予約者には一切表示されない

## 9. デプロイ・環境設定

### 必要な環境変数

| 変数名 | 内容 |
|--------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | SupabaseプロジェクトのURL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabaseの公開APIキー（anon key） |

### Supabaseセットアップ手順（概要）

1. Supabaseでプロジェクトを作成する
2. CLIでリンクする：`supabase link --project-ref <プロジェクトID>`
3. データベースに設計を反映する：`supabase db push`
4. 管理者ユーザーをSupabaseダッシュボード（Authentication > Users）で作成する

### Vercelデプロイ手順（概要）

1. GitHubにリポジトリをプッシュする
2. Vercelにプロジェクトをインポートする
3. 環境変数（NEXT_PUBLIC_SUPABASE_URL、NEXT_PUBLIC_SUPABASE_ANON_KEY）を設定する
4. デプロイを実行する
